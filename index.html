Cloud Modernization

Led a large-scale transition from legacy on‑prem monoliths to a decoupled, cloud‑native architecture. The effort combined database migrations with service boundary redesign, re-writing business logic and database code to fit modern microservices patterns.

AWS RDSAurora (PostgreSQL/Oracle)OracleSQL Server S3RedshiftMicroservicesPL/SQLPython
  • Action: Migrated 50+ Oracle/SQL Server databases to AWS RDS/Aurora, including schema redesign, data validation, and production cutover.
  • Architecture: Split monoliths into service‑oriented and microservices designs by separating service, UI, and SRE layers.
  • Result: Simplified governance and change management; improved incident isolation and reduced operational cost by ~40%.
Design decisions
  • Cutover strategy: staged migrations, dual-run validation, and rollback plans.
  • Schema evolution: versioning, compatibility rules, and conversion patterns.
  • Operability: standard logging/metrics and clear ownership per service boundary.
Reference architecture: monolith split into service/UI/SRE layers → cloud data layer → analytics platforms for insights and governance.
DART pipeline: multi‑source telemetry ingestion → normalization/correlation → anomaly scoring → automated escalation + dashboards.

DART: Real-Time Observability

Built the Digital Alert & Reporting Technology (DART) framework to catch “silent failures” across complex customer journeys. DART correlates error bursts, latency regressions, and conversion drops across tools—then routes alerts to the right owners.

PythonOpenSearchSplunkDynatraceGlassbox SlackSMSGrafana
  • Mechanism: Normalized events across sources; correlated spikes by flow (login → billing → checkout) and deployment windows.
  • Automation: Dynamic SMS/Slack alerting based on application ownership and transaction routing map.
  • Operations: Alert hygiene (dedupe/suppression) + playbooks to reduce noise and speed response.
Key design decisions
  • Normalization layer: platform-specific formats → common event model.
  • Routing map: flow → owner group, escalation path, channel, severity.
  • Noise control: suppression windows and rate limits to prevent alert storms.

Predictive ML: Churn Deciles & Billing Friction

Operationalized ML models that identify churn risk and billing friction at the individual customer level. Outputs were expressed as decile scores to drive consistent playbooks at scale.

PythonPyTorchSQLSnowflakeRedshift Feature EngineeringGovernance
  • Churn model: Decile-based scoring used to target interventions; contributed to >10% churn reduction.
  • Billing friction: Predictive triggers for proactive self-service messaging, reducing support volume by ~20%.
  • Platform: Feature views in Snowflake/Redshift; training/scoring in Python/PyTorch; monitoring for drift and KPI impact.
How it was made usable (beyond accuracy)
  • Deciles mapped directly to retention playbooks and outreach tiers.
  • Outcome tracking focused on uplift and cost-to-serve reduction.
  • Thresholds tuned over time using feedback loops and cohort analysis.
MLOps workflow: warehouse feature views → Python/PyTorch modeling → decile scoring → operational playbooks with governance.

Global Cookie Sharing (GCS)

Designed a centralized session management service to replace brittle cookie-based state across applications. The system improved consistency, reduced coupling, and strengthened security controls around session lifecycle.
